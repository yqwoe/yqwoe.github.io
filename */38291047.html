背景：原来的记忆有什么问题

Hermes Agent 默认的记忆是个 key-value，写进去什么就读出来什么，读取逻辑基本等于字符串匹配。用了一阵子，问题就出来了：

  • "微信截图"和"截图工具"说的是一回事，但搜不到

  • 记忆一多，自己都不知道存哪了

  • 换个 session，上下文全没了

对偶尔用一次的人来说够用，但我的场景是每天几十个任务、跨多个项目，这套机制撑不住。

MemPalace 是什么

MemPalace 是本地语义记忆系统，数据存在 sqlite 里，不依赖任何云服务。组织结构很直接：

  • Wing：项目或领域，比如 hermes-agent、念念

  • Room：维度，比如 general、diary、decisions

  • Drawer：具体的一张记忆卡片，支持语义索引

核心功能四个：

  • mempalace_search — 语义搜索

  • mempalace_kg_query — 知识图谱查询

  • mempalace_add_drawer — 写入记忆

  • mempalace_diary_write — 日记式记录

支持时间线查询，自动归档，用了一阵子感觉比纯 key-value 稳。

改造思路

Hermes Agent 支持 MemoryProvider 插件机制，实现规定接口就能换掉默认记忆层。我写了个 MemPalace Provider，把四个核心工具注册进去，把原来内置的 memory 工具替换掉。

原则就三条：

  1. 读取全走 MemPalace — 收到消息就按需搜，不再靠内置 memory 注入

  2. 写入全走 MemPalace — 项目进展、bug、偏好全部写入，碰都不碰内置 memory

  3. cronjob 自动归档 — 每 30 分钟自动归档 session 进展，不怕断电丢数据

内置 memory 工具现在只留一个职责：存 MemPalace 的路径，其他一概不存。

踩过的坑

1. goja 沙箱限制

最初想在 PocketBase JS Hook 里直接调 MemPalace，结果 PocketBase 0.35+ 用的 goja 对内置对象做了严格限制，node.js 那套基本跑不了。最后改成外面用 Python 脚本中转，Hook 只负责触发。

2. PB 0.37.4 listRule 实测跟文档不符

官方文档写 listRule: null 代表公开读取，实测不是那回事。翻了半天 issue，正确写法是：

有 auth.id 就代表已认证，没登录就没有读取权限。

3. webpack dev server 代理不转发 Authorization

H5 请求打到 PB 后端，webpack dev server 默认不转发 Authorization header，所有认证接口都 401。临时绕法：前端直接请求 PB 地址，绕过代理。

效果

现在每次启动，Agent 能记住：

  • 项目路径、测试账号、用的什么技术栈

  • 上次做到哪、还有什么没搞完

  • 你的偏好，比如截图要整屏不要只截二维码

换 session 不会从头来。搜"微信截图"能找到"截图工具"相关的内容，不同表述不影响。

注意

  • MemPalace 是本地的，没有多设备同步，多台电脑用要自己处理同步

  • 语义搜索质量依赖 embedding 模型，模型选得不好结果会很飘

  • 写入太频繁 sqlite 会越来越肥，隔一阵子得清理一次

结论

从"存了但找不到"变成"语义相近就能找到"。代价是多引入了一个依赖，但换来的是可查询性和可靠性，我觉得值。

如果你是从零做 Agent 记忆系统，建议一开始就想清楚走向量搜索还是纯 key-value，这两条路后面的扩展性差很多。

念念 H5 + PocketBase 还在继续开发，记忆系统只是基础设施之一。后续有新坑再更新。
